로컬에서 코드 짜고 서버에서 실행하는 Git 작업 흐름

이 문서는 현재 프로젝트를 기준으로, Windows 로컬에서 코드를 수정하고 GitHub에 올린 뒤 공용 서버에서 git pull로 최신 코드를 받아 실행하는 방법을 정리한 가이드입니다.

로컬 Windows 코드를 수정하고 테스트한 뒤 GitHub로 올리는 곳입니다.
GitHub 로컬과 서버 사이에서 코드를 주고받는 중간 저장소입니다.
서버 GPU나 채점 환경에 가까운 곳에서 실제 실행하는 곳입니다.

1. 전체 그림

앞으로의 기본 작업 흐름은 아래처럼 생각하면 됩니다.

1
로컬에서 코드 수정

예: 프롬프트 템플릿 수정, retriever 코드 수정, evaluator 보조 스크립트 수정.

2
로컬에서 commit

현재 변경사항을 Git의 기록으로 저장합니다.

3
GitHub에 push

로컬 commit을 GitHub 저장소에 업로드합니다.

4
서버에서 pull

서버가 GitHub에서 최신 코드를 내려받습니다.

5
서버에서 실행

서버 환경에서 evaluator 또는 프로젝트 코드를 돌립니다.

핵심은 commit은 저장, push는 업로드, pull은 다운로드입니다.

2. 현재 프로젝트 상태

현재 로컬 프로젝트는 이미 Git 저장소로 초기화되어 있고, GitHub 저장소와 연결되어 있습니다.

항목 값
GitHub 저장소 https://github.com/stillparsing/dl2026-project
브랜치 main
remote 이름 origin
주요 수정 파일 skeleton/src/solver.py, skeleton/src/document_retriever.py, skeleton/artifacts/prompt_template.md, tools/run_prompt_eval.py

즉, 앞으로는 새로 GitHub 연결을 다시 할 필요 없이 수정, commit, push만 반복하면 됩니다.

3. 로컬 Windows에서 작업하는 방법

3.1 작업 폴더로 이동

PowerShell에서 프로젝트 폴더로 이동합니다.

cd "C:\Users\123wh\Desktop\2026-1\Basics of Deep Learning\project"

3.2 변경사항 확인

어떤 파일이 바뀌었는지 먼저 봅니다.

& "C:\Program Files\Git\cmd\git.exe" status

짧게 보고 싶으면: 

& "C:\Program Files\Git\cmd\git.exe" status --short

3.3 변경한 파일을 Git에 올릴 준비

모든 변경사항을 한 번에 올릴 준비를 하려면: 

& "C:\Program Files\Git\cmd\git.exe" add .

특정 파일만 올리고 싶으면 파일 경로를 직접 적습니다. 

& "C:\Program Files\Git\cmd\git.exe" add skeleton/artifacts/prompt_template.md

3.4 commit 만들기

commit은 "이번 변경사항 묶음"에 이름을 붙여 저장하는 단계입니다. 메시지는 너무 길 필요 없고, 무엇을 바꿨는지만 알 수 있으면 됩니다. 

& "C:\Program Files\Git\cmd\git.exe" commit -m "Update prompt template"

예시 commit 메시지:

  • Update prompt template
  • Improve document retrieval scoring
  • Add prompt evaluation helper
  • Fix evaluator path handling

3.5 GitHub에 push

commit을 GitHub에 업로드합니다. 

& "C:\Program Files\Git\cmd\git.exe" push
로컬에서 수정만 하고 push하지 않으면 서버는 그 변경사항을 받을 수 없습니다. 서버는 GitHub에 올라간 코드만 git pull로 가져옵니다.

4. 서버에서 처음 한 번만 하는 일

공용 서버에서는 다른 사람 파일과 섞이지 않도록 본인 계정의 홈 디렉토리 아래에 프로젝트 폴더를 따로 만드는 것이 좋습니다. 

mkdir -p ~/projects
cd ~/projects
git clone https://github.com/stillparsing/dl2026-project.git
cd dl2026-project

이렇게 하면 서버에는 아래 위치에 프로젝트가 생깁니다. 

~/projects/dl2026-project
공용 서버의 /tmp, /shared, /home/common 같은 곳에 바로 프로젝트를 두는 것은 피하세요. 가능하면 본인 홈 디렉토리 아래에 둡니다.

5. 서버에서 매번 업데이트하는 방법

서버에 이미 프로젝트를 clone해둔 뒤에는, 매번 다시 clone할 필요가 없습니다. 로컬에서 push한 변경사항을 서버에서 pull하면 됩니다. 

cd ~/projects/dl2026-project
git pull

이 명령은 GitHub에 올라간 최신 commit을 서버 폴더로 가져옵니다.

서버에서 git pull을 했는데 바뀐 내용이 없다면, 로컬에서 아직 commit 또는 push를 하지 않았을 가능성이 큽니다.

6. 서버에서 실행하는 방법

프로젝트 구조상 실제 evaluator는 skeleton 폴더 안에 있습니다. 서버에서 코드를 받은 뒤 아래처럼 이동합니다. 

cd ~/projects/dl2026-project/skeleton

처음 실행 환경을 잡아야 한다면 README 기준으로 setup을 먼저 실행합니다. 

bash setup.sh

그 다음 evaluator를 실행합니다. 

python evaluate.py

우리가 만든 보조 실행 스크립트를 쓰려면 프로젝트 루트 기준으로 실행합니다. 

cd ~/projects/dl2026-project
python tools/run_prompt_eval.py

데이터셋 경로가 서버에서 다르면 evaluator나 스크립트가 요구하는 방식에 맞춰 경로를 지정해야 합니다. 이 부분은 서버의 과제 안내, 조교 제공 evaluator 형식, 실제 데이터 위치에 따라 달라질 수 있습니다.

7. 자주 쓰는 명령어 요약

로컬 Windows

cd "C:\Users\123wh\Desktop\2026-1\Basics of Deep Learning\project"
& "C:\Program Files\Git\cmd\git.exe" status
& "C:\Program Files\Git\cmd\git.exe" add .
& "C:\Program Files\Git\cmd\git.exe" commit -m "Update prompt"
& "C:\Program Files\Git\cmd\git.exe" push

서버

cd ~/projects/dl2026-project
git pull
cd skeleton
python evaluate.py

8. 실수했을 때 보는 체크리스트

서버에서 최신 코드가 안 보일 때

  1. 로컬에서 git status를 확인합니다.
  2. 수정 파일이 있으면 git add .를 했는지 확인합니다.
  3. git commit -m "메시지"를 했는지 확인합니다.
  4. git push를 했는지 확인합니다.
  5. 서버에서 올바른 폴더에서 git pull을 했는지 확인합니다.

git commit에서 "nothing to commit"이 나올 때

Git 입장에서는 저장할 변경사항이 없다는 뜻입니다. 이미 commit했거나, 파일을 실제로 저장하지 않았거나, 변경 파일이 .gitignore에 의해 제외되었을 수 있습니다.

git pull에서 충돌이 날 때

서버에서도 직접 파일을 수정했고, 로컬에서도 같은 파일을 수정한 경우 충돌이 날 수 있습니다. 지금 작업 방식에서는 서버에서는 되도록 코드를 직접 수정하지 않고 실행만 하는 것이 안전합니다. 코드는 로컬에서 수정하고 서버는 pull만 하는 방식으로 유지하세요.

GitHub 로그인 창이 뜰 때

git push 중 Git Credential Manager가 뜨면 GitHub 계정으로 로그인하면 됩니다. 한 번 인증하면 보통 다음부터는 다시 묻지 않습니다.

9. 이 프로젝트에서 Git에 올리지 말아야 할 것

코드와 프롬프트는 GitHub에 올려도 되지만, 무거운 파일이나 개인 작업 파일은 올리지 않는 것이 좋습니다. 현재 .gitignore에 주요 제외 항목을 넣어두었습니다.

  • dataset/: 공개/비공개 데이터셋은 보통 Git에 올리지 않습니다.
  • project.pdf.pdf: 과제 PDF는 저장소에 올리지 않도록 제외했습니다.
  • progress.md: 대화 요약 파일은 사용자가 요청할 때만 저장하고, Git에는 올리지 않습니다.
  • 모델 weight, cache, log, score, prediction 파일: 크거나 재생성 가능한 결과물이므로 Git에 올리지 않습니다.
특히 서버에서 생성된 결과 파일을 무심코 git add .로 올리지 않도록 주의하세요. 이상하면 먼저 git status로 어떤 파일이 올라갈지 확인합니다.

10. 가장 추천하는 습관

  1. 코드는 로컬에서 수정합니다.
  2. 서버에서는 가능하면 직접 코드를 수정하지 않습니다.
  3. 로컬에서 작은 단위로 commit합니다.
  4. 실험 하나가 끝나면 push합니다.
  5. 서버에서는 실행 전 항상 git pull을 합니다.
  6. 이상한 파일이 Git에 잡히면 commit하기 전에 멈추고 확인합니다.

이 흐름에 익숙해지면 로컬은 편하게 개발하는 곳, GitHub는 코드 전달 창구, 서버는 실행하는 곳으로 역할이 분리됩니다. 그러면 서버 환경이 조금 불편해도 로컬에서 빠르게 고치고 서버에서 바로 검증할 수 있습니다.
