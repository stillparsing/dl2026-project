1. 한 줄 요약

현재 제출 버전은 규칙 기반 상태 추적 + 보수적인 deterministic prefilter + Qwen fallback을 결합한 hybrid solver다. 공개 예제와 우리가 만든 synthetic set에서는 강하게 맞췄지만, hidden test에서는 62.5점에 그쳐 일반화 성능이 충분하지 않았다.

핵심 판단 기준은 “마지막 target response가 이전 trajectory로 형성된 상태에서 protocol상 가능한 응답인가?”이다. 이전 step들은 정답 여부가 아니라 target 직전 상태를 계산하기 위한 context로만 사용한다.

2. 전체 파이프라인

1. Raw testcase JSON trajectory를 읽는다. 각 step은 method command 또는 data read/write command다.
2. 압축 긴 UID, secret, binary-like string을 alias로 줄이고 target step을 명확히 표시한다.
3. State ledger 성공한 이전 step만 상태에 반영한다. 실패한 이전 step은 state change로 보지 않는다.
4. Prefilter 확실한 pass/fail만 deterministic rule로 즉시 판정한다.
5. Qwen fallback 애매한 케이스는 compact prompt와 관련 spec snippet을 넣어 Qwen이 최종 판단한다.

3. 주요 파일과 역할

파일 역할
skeleton/src/solver.py Solver entrypoint. Rule prefilter를 먼저 실행하고, 필요할 때만 Qwen 모델을 lazy loading한다.
skeleton/src/preprocess.py Trajectory 압축, state ledger 생성, target cue 생성, deterministic prefilter rule을 담당한다.
skeleton/src/document_retriever.py Target step과 관련된 TCG/Opal spec snippet을 artifacts 문서에서 검색한다.
skeleton/artifacts/prompt_template.md Qwen fallback에 들어가는 판단 지침과 calibration rule을 담는다.
skeleton/setup.sh 제출 환경에서 의존성을 설치하고 Qwen base model snapshot을 미리 다운로드한다.

4. State ledger 아이디어

LLM에게 전체 raw JSON을 그대로 읽히면 작은 모델은 긴 trajectory에서 핵심 상태를 놓치기 쉽다. 그래서 preprocess.py가 target 직전 상태를 구조화된 ledger로 만든다.

추적하는 상태

  • active_session: 현재 session 존재 여부, SP, authority, write flag
  • sp_lifecycle: 특히 LockingSP activation 여부
  • authorities: SID/Admin1/User1 관련 pin 변경, authority enable 관찰
  • locking: Locking range, MBRControl 관찰/수정
  • crypto: GenKey event, data write 이후 GenKey 발생 여부
  • data_path: Write/Read 결과

상태 업데이트 원칙

  • 이전 step은 순서대로 처리한다.
  • 성공한 method/data step만 상태를 바꾼다.
  • 실패한 이전 method response는 관찰값일 뿐 state change가 아니다.
  • EndSession 성공은 active session을 없앤다.
  • 마지막 target step은 state에 적용하지 않고, 그 직전 상태와 비교한다.

5. Deterministic prefilter

Prefilter는 hidden generalization을 망칠 수 있으므로, “거의 확실한 visible pattern”만 처리하도록 보수적으로 유지했다. 그래도 synthetic set에서 Qwen이 반복적으로 틀리는 패턴은 좁은 조건으로 rule화했다.

대상 현재 rule의 예 의도
Properties 정상 HostProperties 요청 + SUCCESS + return values → pass 기본 negotiation 성공 패턴 처리
Properties 명백히 malformed Properties + INVALID_PARAMETER → pass 올바른 거절 응답도 pass일 수 있음을 처리
Properties 정상 Properties 요청 + error status → fail Qwen의 “error도 pass” 과보정을 막음
StartSession LockingSP가 activation 전인데 SUCCESS → fail 상태와 모순되는 성공 응답 제거
StartSession LockingSP activation 전 거절 응답 + no session IDs → pass 정상 거절 응답 처리
StartSession HostChallenge에 non-alphanumeric separator가 있고 거절됨 → pass 명백한 malformed token 거절 처리
Get/Set/GenKey active session 없이 SUCCESS → fail session이 필요한 command의 잘못된 성공 처리
Get/Set/GenKey active session 없이 NOT_AUTHORIZED → pass 올바른 authorization rejection 처리
Read after GenKey data write 이후 GenKey가 있고 Random Data → pass, old/plain data → fail data path randomization 패턴 처리

6. Qwen fallback

Prefilter가 결론을 내지 못하면 Qwen이 판단한다. 이때 raw trajectory 대신 압축된 case summary, target judgment focus, state ledger, 관련 spec snippet을 prompt에 넣는다.

  • 모델은 Solver.__init__에서 바로 로드하지 않고, fallback이 필요한 첫 케이스에서 lazy loading한다.
  • target_judgment_focus.expected_response_hint는 target이 accept되어야 하는지 reject되어야 하는지를 짧게 알려준다.
  • prompt는 반드시 소문자 pass 또는 fail 한 단어만 내도록 제한한다.
  • 생성 결과에서 pass|fail token을 regex로 추출하고, 실패 시 기본값은 fail이다.

8. 현재 한계

  • Hidden test 점수가 62.5점으로 낮다. 즉 synthetic set이 실제 hidden 분포를 충분히 대표하지 못했다.
  • Prefilter는 public/synthetic 패턴에 맞춰진 부분이 있어 hidden에서 false positive/false negative 위험이 있다.
  • Qwen fallback은 여전히 약하다. 특히 prefilter가 커버하지 않는 ACL, SP/authority 세부 규칙, data path 변형에 취약할 가능성이 높다.
  • Spec retrieval은 단순 관련 snippet 검색이므로, 실제 판단에 필요한 rule card를 안정적으로 제공하지 못할 수 있다.
  • 현재 방식은 testcase별 독립 판단에 가깝고, hidden adversarial family를 체계적으로 생성/검증하는 과정이 부족했다.

9. 다음 개선 방향

  • Hidden score를 올리려면 public case를 변형하는 synthetic만으로는 부족하다. spec 기반 adversarial generator를 더 넓게 만들어야 한다.
  • Prefilter rule은 “상태상 확정 가능한 expected behavior”를 계산하는 작은 interpreter 형태로 정리하는 것이 좋다.
  • Qwen prompt는 긴 자연어 경고보다 구조화된 rule card를 우선해야 한다.
  • 실제 채점 실패 유형을 알 수 없다면, 여러 제출을 통해 score 변화와 rule ablation을 비교해야 한다.
  • 가능하다면 Qwen fallback 의존도를 줄이고, public spec에서 반복되는 state machine/ACL 조건을 deterministic하게 더 구현해야 한다.
