1. 큰 흐름

초기에는 Qwen에게 압축된 testcase와 spec snippet을 주고 pass/fail을 직접 판단하게 하는 prompt 중심 접근이었다. 하지만 public/synthetic case에서 작은 모델이 stateful reasoning을 자주 놓쳤고, 이후 deterministic state ledger와 prefilter를 결합한 hybrid 접근으로 바뀌었다.

마지막에는 synthetic v1/v2를 모두 맞췄지만, 실제 hidden score는 62.5점이었다. 이는 public-derived synthetic에 과하게 맞춘 규칙과, hidden distribution을 충분히 반영하지 못한 adversarial generation의 한계를 보여준다.

State ledger 도입
71f8f34

Deterministic state preprocessing 추가

state ledger preprocess
  • preprocess.py가 이전 step을 순서대로 처리해 target 직전 상태를 만든다.
  • 성공한 이전 step만 state change로 반영하고, 실패한 이전 method response는 관찰만 한다.
  • active session, LockingSP activation, authority/PIN 상태, locking range, GenKey/data path를 추적했다.
  • Prompt에는 state_before_target, state_update_trace, target_judgment_focus가 들어가도록 했다.

이 단계부터 “LLM이 상태를 처음부터 추론하게 하지 말고, 전처리가 상태를 요약해준다”는 방향으로 바뀌었다.

Accepted error response 테스트 추가

adversarial correct rejection
  • tools/generate_adversarial_cases.py를 추가했다.
  • “명령이 거절됐으므로 fail”이 아니라 “상태상 거절되어야 하므로 pass”인 케이스를 만들었다.

처음에는 Qwen이 accepted error response를 거의 전부 fail로 찍었다.

v2 보정 단계
8ec08b8

엄격한 malformed rejection rule 추가

rules careful
  • Qwen이 계속 틀리는 v2의 tc2005, tc2006을 분석했다.
  • 처음에는 malformed HostChallenge를 넓게 rule로 넣었지만 v1에서 false positive가 발생해 되돌렸다.

이 단계에서 “테스트를 쉽게 만드는 것”이 아니라 “라벨과 입력이 일치하도록 synthetic case를 공정하게 만드는 것”을 구분했다.

4. 배운 점

잘 된 점

  • 상태 ledger는 작은 LLM의 약점을 보완하는 데 효과적이었다.
  • Prefilter audit 도구는 rule false positive를 빠르게 잡는 데 유용했다.
  • Accepted error response를 별도 adversarial set으로 만든 것은 중요한 방향이었다.
  • Lazy loading으로 prefiltered case 처리 속도를 개선했다.

부족했던 점

  • Synthetic case가 public example 변형에 너무 의존했다.
  • Hidden leaderboard를 설명할 만큼 다양한 protocol/ACL edge case를 만들지 못했다.
  • Prompt 문구 강화는 임시 보정에 가깝고, 안정적인 구조화 판단으로 충분히 전환하지 못했다.
  • Spec retrieval이 실제 decision rule로 변환되지 않아 Qwen fallback의 품질이 낮았다.

5. 다음 작업 후보

  • Hidden score 개선을 위해 prefilter rule을 늘리기 전에, rule audit용 adversarial set을 훨씬 넓혀야 한다.
  • StartSession, Get, Set, Activate, GenKey별로 expected accept/reject를 계산하는 명시적 rule card를 만들 필요가 있다.
  • Authority/ACL 관계를 더 정확히 모델링해야 한다. 현재는 Admin1/User1/SID의 일부 패턴만 추적한다.
  • Data path는 GenKey 여부만 보는 수준에서 벗어나 locking range, read/write enable, MBR 상태와 연결해야 한다.
  • Qwen fallback은 최종 보조 수단으로 두고, 핵심 state machine은 deterministic하게 구현하는 방향이 더 현실적이다.
다음 제출을 준비한다면 단순 prompt 수정만으로 큰 점수 상승을 기대하기 어렵다. hidden score 62.5점은 “LLM fallback보다 protocol state interpreter를 더 많이 구현해야 한다”는 신호로 보는 것이 타당하다.
